IoT Communication Protocols - Complete Comparison and Reference

=== MQTT vs CoAP vs HTTP COMPARISON ===

Feature Comparison Table:
Protocol     | Transport | Pattern    | QoS    | Overhead | Best For
-------------|-----------|------------|--------|----------|------------------
MQTT         | TCP       | Pub/Sub    | 0,1,2  | Low      | Real-time telemetry
CoAP         | UDP       | Req/Resp   | Non/Con| Very Low | Constrained devices
HTTP/REST    | TCP       | Req/Resp   | None   | High     | Cloud APIs, dashboards
WebSocket    | TCP       | Bi-dir     | None   | Medium   | Real-time web UIs
AMQP         | TCP       | Pub/Sub    | High   | High     | Enterprise messaging
STOMP        | TCP       | Pub/Sub    | Basic  | Medium   | Simple broker access

MQTT Strengths:
- Perfect for unreliable networks (handles disconnect/reconnect gracefully)
- Low bandwidth and battery consumption
- Pub/Sub decouples publishers from subscribers
- QoS levels ensure reliable delivery when needed
- LWT (Last Will) for device status management
- Retained messages for state persistence

MQTT Weaknesses:
- Requires always-on broker
- No built-in request/response pattern (workaround with response topics)
- Security depends on TLS layer (TCP-based)
- Not ideal for synchronous query/response patterns

CoAP Strengths:
- Runs over UDP, lower overhead than TCP
- Designed for constrained environments (RFC 7252)
- RESTful semantics (GET, PUT, POST, DELETE)
- Supports multicast for group communication
- Observable resources (subscribe to resource changes)
- Built-in reliability via Confirmable messages

CoAP Weaknesses:
- UDP can lose packets on poor networks
- Less ecosystem support than MQTT
- No built-in pub/sub (must use observe extension)
- Requires specific CoAP libraries

HTTP/REST Strengths:
- Universal support across all languages and platforms
- Stateless - easy to scale horizontally
- Familiar to web developers
- Works through most firewalls and proxies
- JSON/XML data formats universally understood

HTTP/REST Weaknesses:
- High overhead (headers, TCP handshake)
- Not suitable for battery-powered devices
- Polling required for "push" behavior (wasteful)
- Latency higher than MQTT for real-time data

When to use each:
- MQTT: Sensor telemetry, device command-and-control, real-time monitoring
- CoAP: Ultra-constrained devices (8-bit MCUs), mesh networks
- HTTP: Cloud API integration, dashboards, webhooks, one-time configuration

=== LoRaWAN ===

Overview:
LoRaWAN (Long Range Wide Area Network) is a low-power wide-area network protocol built on LoRa (Long Range) radio modulation. It enables long-range communication (2-15 km in urban, up to 50 km rural line-of-sight) at very low power consumption.

Key Specifications:
- Frequency bands: 868 MHz (Europe), 915 MHz (Americas), 433 MHz, 923 MHz (Asia)
- Data rate: 0.3 kbps to 50 kbps (spreading factor dependent)
- Range: 2-15 km urban, up to 50 km LoS
- Battery life: Up to 10 years on AA batteries
- Maximum payload: 51-222 bytes (depending on spreading factor)
- Bidirectional: Yes (Class A, B, C devices)
- Network topology: Star-of-stars (devices → gateways → network server)

LoRaWAN Classes:
Class A (All devices must support):
- Uplink anytime, two short downlink windows after each uplink
- Lowest power consumption
- Best for sensors that send periodic data

Class B (Beacon-based):
- Synchronized downlink windows (beaconed from gateway)
- Moderate power consumption
- Good for actuators needing predictable latency

Class C (Continuous receive):
- Continuous downlink listening (no scheduled windows)
- Highest power consumption (unsuitable for battery)
- Good for mains-powered actuators

Spreading Factors (SF7-SF12):
SF7: Fastest, shortest range, lowest battery life
SF12: Slowest, longest range, best sensitivity (-137 dBm), most battery use

Arduino LoRaWAN Example (LMIC library):
#include <lmic.h>
#include <hal/hal.h>
#include <SPI.h>

// Network credentials (from TTN console)
static const u1_t PROGMEM APPEUI[8] = {0x00, ...};
static const u1_t PROGMEM DEVEUI[8] = {0x00, ...};
static const u1_t PROGMEM APPKEY[16] = {0x00, ...};

void os_getArtEui(u1_t* buf) { memcpy_P(buf, APPEUI, 8); }
void os_getDevEui(u1_t* buf) { memcpy_P(buf, DEVEUI, 8); }
void os_getDevKey(u1_t* buf) { memcpy_P(buf, APPKEY, 16); }

void do_send(osjob_t* j) {
    uint8_t payload[2];
    int16_t temp = (int16_t)(readTemperature() * 100);
    payload[0] = temp >> 8;
    payload[1] = temp & 0xFF;

    LMIC_setTxData2(1, payload, sizeof(payload), 0);
}

Popular LoRaWAN Networks:
- The Things Network (TTN): Free, community-run global network
- Helium Network: Decentralized, crypto-incentivized
- AWS IoT Core for LoRaWAN: Amazon managed service
- Private LoRaWAN: Self-hosted ChirpStack network server

=== BLUETOOTH LOW ENERGY (BLE) ===

Overview:
BLE (Bluetooth Low Energy, also Bluetooth Smart) is a wireless technology for short-range, low-power communication. Introduced in Bluetooth 4.0, optimized for IoT applications requiring long battery life.

Key Specifications:
- Range: 10-100 meters (typically 50m indoor)
- Frequency: 2.4 GHz ISM band (40 channels)
- Data rate: 1 Mbps (BLE 4.x), 2 Mbps (BLE 5.0), 125 kbps / 500 kbps (BLE 5.0 Long Range)
- Latency: 3ms minimum connection interval
- Power: 0.01-0.50 W (vs Classic BT 1W)
- Maximum connections: Technically unlimited, practically 8-20 per central

BLE Concepts:
GAP (Generic Access Profile): Controls device discovery and connection
GATT (Generic Attribute Profile): Defines data exchange (services and characteristics)
Peripheral: Advertises and waits for connection (sensor nodes)
Central: Scans and initiates connection (phone, gateway)
Service: Collection of related characteristics (e.g., Heart Rate Service)
Characteristic: Data value with properties (Read, Write, Notify, Indicate)
Descriptor: Metadata about a characteristic

Common BLE Services (standard UUIDs):
0x180D - Heart Rate Service
0x180A - Device Information Service
0x1800 - Generic Access Service
0x1801 - Generic Attribute Service
0x181A - Environmental Sensing Service (temperature, humidity)
0x1816 - Cycling Speed and Cadence

ESP32 BLE Example (Arduino):
#include <BLEDevice.h>
#include <BLEServer.h>
#include <BLEUtils.h>
#include <BLE2902.h>

#define SERVICE_UUID        "4fafc201-1fb5-459e-8fcc-c5c9c331914b"
#define CHARACTERISTIC_UUID "beb5483e-36e1-4688-b7f5-ea07361b26a8"

BLEServer* pServer = nullptr;
BLECharacteristic* pCharacteristic = nullptr;
bool deviceConnected = false;

class ServerCallbacks : public BLEServerCallbacks {
    void onConnect(BLEServer* pServer) { deviceConnected = true; }
    void onDisconnect(BLEServer* pServer) {
        deviceConnected = false;
        pServer->startAdvertising();
    }
};

void setup() {
    BLEDevice::init("ESP32-Sensor");
    pServer = BLEDevice::createServer();
    pServer->setCallbacks(new ServerCallbacks());

    BLEService* pService = pServer->createService(SERVICE_UUID);
    pCharacteristic = pService->createCharacteristic(
        CHARACTERISTIC_UUID,
        BLECharacteristic::PROPERTY_READ | BLECharacteristic::PROPERTY_NOTIFY
    );
    pCharacteristic->addDescriptor(new BLE2902());
    pService->start();

    BLEAdvertising* pAdvertising = BLEDevice::getAdvertising();
    pAdvertising->addServiceUUID(SERVICE_UUID);
    pAdvertising->setScanResponse(true);
    BLEDevice::startAdvertising();
}

void loop() {
    if (deviceConnected) {
        float temp = 23.5;
        pCharacteristic->setValue(temp);
        pCharacteristic->notify();
        delay(1000);
    }
}

BLE Advertising (iBeacon/Eddystone):
BLE devices can broadcast data without pairing via advertising packets.
Used for: Asset tracking, proximity detection, indoor positioning.

BLE 5.0 Improvements:
- 2x speed (2 Mbps mode)
- 4x range (coded PHY long range mode)
- 8x advertising data capacity (Extended Advertising)
- Direction finding (AoA/AoD for ~cm positioning)

=== ZIGBEE ===

Overview:
Zigbee is an IEEE 802.15.4-based specification for a suite of high-level communication protocols for personal area networks. Designed for low-power mesh networking with dozens to hundreds of nodes.

Key Specifications:
- Frequency: 2.4 GHz (global), 868 MHz (Europe), 915 MHz (Americas)
- Data rate: 250 kbps (2.4 GHz)
- Range: 10-100 meters per hop
- Mesh range: Effectively unlimited with enough routers
- Power: Very low, coin cell battery lasts years
- Network size: Up to 65,535 nodes theoretically
- Latency: ~15ms minimum

Zigbee Network Roles:
Coordinator: Forms and manages network (one per network)
Router: Relays messages, always-on, extends mesh range
End Device: Leaf node, can sleep, minimal function

Zigbee Device Types in Smart Home:
- Philips Hue bulbs, IKEA Tradfri lights
- Aqara sensors (temperature, door, motion)
- Xiaomi sensors
- Sonoff Zigbee devices
- Amazon Echo (Zigbee coordinator built-in)

Zigbee vs Z-Wave vs Thread:
Feature    | Zigbee        | Z-Wave       | Thread
-----------|---------------|--------------|------------------
Frequency  | 2.4 GHz       | 908 MHz      | 2.4 GHz
Protocol   | 802.15.4      | Proprietary  | 802.15.4 (IPv6)
Mesh       | Yes           | Yes          | Yes
Nodes      | 65535         | 232          | Unlimited
Interop    | Profile-based | Yes (cert)   | Matter standard
Power      | Ultra-low     | Ultra-low    | Ultra-low
Used by    | SmartThings   | Ring Alarm   | Google, Apple

=== MATTER (formerly Project CHIP) ===

Overview:
Matter is an open-source, royalty-free smart home connectivity standard backed by Apple, Google, Amazon, Samsung, and 200+ companies. Built on IPv6 and Thread/Wi-Fi.

Key Features:
- Runs on Thread (mesh) or Wi-Fi
- Local control - works without internet
- Compatible across Apple Home, Google Home, Amazon Alexa, Samsung SmartThings
- Based on IPv6 - every device has an IP address
- End-to-end encryption required

=== PROTOCOL SELECTION GUIDE ===

Long range, low power, rural/outdoor: LoRaWAN
Short range, very low power, coin cell: Zigbee or BLE
Real-time telemetry over internet: MQTT
REST API integration: HTTP
Local mesh smart home: Zigbee, Z-Wave, or Thread/Matter
Local proximity (phone to device): BLE
Video streaming: WebRTC, RTSP
Industrial automation: Modbus, OPC-UA, PROFINET
Building automation: BACnet, KNX
